# docker volume

## voulmn 
docker에서 사용하는 저장소는 크게 3가지가 있음.
컨테이너 밖에 데이터를 저장하는 공간으로, 컨테이너가 삭제되어도 데이터가 유지될 수 있음.
즉. "실행 환경"과 "데이터"를 분리하는 개념.
aws에서는 EBS.

| 종류               | 저장 위치                                        | 주 사용 목적          |
| ---------------- | -------------------------------------------- | ---------------- |
| **Bind Mount**   | 호스트의 특정 디렉터리 (`~/html`)                      | 개발, 설정 파일 공유     |
| **Named Volume** | Docker가 관리하는 저장소 (`/var/lib/docker/volumes`) | 운영 환경에서 데이터 영속성  |
| **tmpfs Mount**  | 메모리(RAM)                                     | 임시 데이터, 매우 빠른 작업 |


### docker volume의 필요성
컨테이너 : 이미지 + 읽고 쓰기가 가능한 레이어
                -> 이 레이어에서 수정하는 것.
컨테이너가 삭제되고 다시 실행되면, 컨테이너 내부에서 수정했던 내용들이 전부 사라짐.
-> 컨테이너는 일시적인 "실행 환경"

## 1. bind mount volume

### bind mount란?
```bash
EC2 (Host)
│
├── ~/html -> ec2에 별도 폴더를 만듦
│      └── index.html
│
└── Docker Engine
       │
       └── nginx Container -> 컨테이너는 일회성
              │
              └── /usr/share/nginx/html
                  -> 컨테이너가 삭제되면 이 공간에서 수정한 내용도 전부 삭제됨
```
두 폴더를 연결하는 것.
host의 "~/html"와 컨테이너의 "/usr/share/nginx/html"를 하나처럼 사용하게 함.
엄밀하게 보면, host 디렉토리를 container가 빌려쓰는 느낌. (host 중심)
그치만 host, container 에서에서 수정해도 수정내역이 같이 보임.

### volume mount해서 container 실행하기
mount해서 컨테이너는 실행하면, host에 있는 html 파일을 수정하면 컨테이너 안의 html에서도 동일한 수정 내역 확인 가능. -> 같은 파일을 보고 있기 때문
```bash
# 명령어
sudo docker run -d \
--name web \
-p 80:80 \
-v ~/html:/usr/share/nginx/html \
nginx
```

```bash
Host (~/html) ────┐
                  │ mount
                  ▼
Container (/usr/share/nginx/html)
```

### mount 확인

```bash
# 명령어
docker inspect web

# 결과
# Source의 있는 파일을 Destination에서 보고 있음.
"Mounts": [
    {
        "Type": "bind", -> 타입 잘 확인하기!
        "Source": "/home/ubuntu/html",
        "Destination": "/usr/share/nginx/html",
        "Mode": "",
        "RW": true,
        "Propagation": "rprivate"
    }
],
```

## 2.named volume
### named volume 이란?

실무에서 bind mount 보다 많이 사용되는 mount
db 데이터는 대부붐 named volume에 저장하고 있음.

- bind volume : host의 디렉토리를 직접 지정
- named volume : host의 경로를 직접 지정하지 않고 docker가 알아서 관리
                 host를 직접 건드릴 일이 없음.

1. volume 생성
경로를 지정하지 않고 볼륨을 생성했음.
도커가 알아서 "/var/lib/docker/volumes/" 경로에 생성함

볼륨 정보 확인 시 "Mountpoint" 경로가 도커가 실제 데이터를 저장하는 위치
```bash
# 볼륨 생성
docker volume create nginx-data

# 볼륨 생성 확인
docker volumn ls

# 볼륨 정보 확인
docker volume inspect nginx-data

##결과
[
    {
        "CreatedAt": "2026-07-16T02:35:36Z",
        "Driver": "local",
        "Labels": null,
        "Mountpoint": "/var/lib/docker/volumes/nginx-data/_data", -> 경로!
        "Name": "nginx-data",
        "Options": null,
        "Scope": "local"
    }
]
```

2. named volume mount해서 컨테이너 실행

```bash
docker run -d \
--name web \
-p 80:80 \
-v nginx-data:/usr/share/nginx/html \
nginx
```

3. mount 확인
named volume으로 Mount하면 type이 달라짐.

```bash
# 명렁어
docker inspect web

# 결과
"Mounts": [
    {
        "Type": "volume", -> bind volume와 type이 다름.
        "Name": "nginx-data",
        "Source": "/var/lib/docker/volumes/nginx-data/_data",
        "Destination": "/usr/share/nginx/html",
        "Driver": "local",
        "Mode": "z",
        "RW": true,
        "Propagation": ""
    }
],
```


## 3.tmpfs volume
### tmpfs volume 이란?
디스크에 저장하지 않고 RAM에 저장하는 저장소
메모리에 저장하기 때문에 빠르고, 재부팅 시 데이터가 유지되지 않음
비밀번호, 세션, 암호키 등 디스크에 남기고 싶지 않을 때 사용

1. tmpfs mount해서 컨테이너 실행
-v 옵션이 아니라 --tmpfs 옵션을 사용
```bash
docker run -d \
--name tmp-nginx \
-p 8080:80 \
--tmpfs /usr/share/nginx/html \
nginx
```
